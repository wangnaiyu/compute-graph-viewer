Step 02 · 参数设计

最小可用参数集：blockLength、tileNum、tileLength

入门阶段先建立可验证的切分模型：每个 core 处理的数据长度、核内切分数量、每个 tile 的处理长度。复杂算子可以继续扩展 cost model，但应保持这些基础字段的可追踪性。

blockLength totalLength / blockDim 每个 core 负责的数据长度。
tileNum kernel loop count 每个 core 内部切成多少块。
tileLength blockLength / tileNum 一次 CopyIn / Compute / CopyOut 处理的长度。
Add 入门切分 · conceptual
TOTAL_LENGTH = 8 * 2048 blockDim = 8 blockLength = 2048 tileNum = 16 tileLength = 128 gmOffset = blockIdx * blockLength + progress * tileLength
边界要求：totalLength 不能整除 blockDim 或 tileNum 时，需要明确尾块策略；tileLength 不能满足 data block / vector 对齐时，需要明确 mask 或 padding 策略。
Step 03 · TilingData

将 Kernel 所需标量参数定义为 TilingData

TilingData 是 Host 侧计算结果和 Kernel 侧执行逻辑之间的合同。字段应保持最小化，同时覆盖切分、循环、搬运和边界处理。

字段定义

字段通常包括总长度、每核长度、核内 tile 数、每 tile 长度、尾块长度、使用核数等。

结构体注册

注册用于供框架识别该 op 使用的 TilingData 结构；涉及 key 分支时，还需要处理默认结构和 key 匹配结构。

op_host/add_custom_tiling.h · conceptual
namespace optiling { BEGIN_TILING_DATA_DEF(AddCustomTilingData) TILING_DATA_FIELD_DEF(uint32_t, totalLength); TILING_DATA_FIELD_DEF(uint32_t, blockLength); TILING_DATA_FIELD_DEF(uint32_t, tileNum); TILING_DATA_FIELD_DEF(uint32_t, tileLength); END_TILING_DATA_DEF; REGISTER_TILING_DATA_CLASS(AddCustom, AddCustomTilingData) }
命名要求：字段名应直接表达 Kernel 侧用途，例如 blockLength、tileLength、tailLength；避免使用 a/b/field1 这类抽象名称。

ge::graphStatus TilingFunc(gert::TilingContext* context) { auto shape = context->GetInputShape(0)->GetStorageShape(); uint32_t totalLength = CalcTotalLength(shape); AddCustomTilingData tiling; tiling.set_totalLength(totalLength); tiling.set_blockLength(CalcBlockLength(totalLength, blockDim)); tiling.set_tileNum(tileNum); tiling.set_tileLength(CalcTileLength(...)); context->SetBlockDim(blockDim); tiling.SaveToBuffer(context->GetRawTilingData()->GetData(), context->GetRawTilingData()->GetCapacity()); context->GetRawTilingData()->SetDataSize(tiling.GetDataSize()); return ge::GRAPH_SUCCESS; }
Step 05 · Kernel 侧使用

Kernel 侧按 TilingData 控制搬入、计算和搬出

Kernel 解析 TilingData 后，把 blockIdx 和循环 progress 映射成 GM offset。CopyIn / Compute / CopyOut 是 tile 生命周期，不是三个孤立概念。

Kernel 使用 tiling · conceptual
GET_TILING_DATA(tilingData, tiling); for (uint32_t progress = 0; progress < tilingData.tileNum; ++progress) { uint32_t gmOffset = blockIdx * tilingData.blockLength + progress * tilingData.tileLength; CopyIn(gmOffset, tilingData.tileLength); Compute(tilingData.tileLength); CopyOut(gmOffset, tilingData.tileLength); }

CopyIn

从 GM 的当前 slice 搬到 Local Memory。关键参数是 gmOffset、length 和对齐。

Compute

在 Vector/Cube pipeline 上消费本地 tensor。关键参数是 tileLength、mask 和 dtype。

CopyOut

把本地结果写回 GM 的对应 slice。关键参数是输出 offset 和尾块。

可视化检查项：当前 step 的 gmOffset 是否正确、Local Memory 是否放得下、队列是否按预期生产/消费、尾块是否被正确处理。
Step 06 · Dynamic Shape / TilingKey

动态 shape 下，TilingKey 是 Host 与 Kernel 约定的分支编号

如果一个 TilingData 结构和一套 kernel 逻辑无法覆盖所有 shape/dtype/attr，就需要 key。Host 侧设置 key，Kernel 侧用相同 key 选择对应分支或结构体。

不需要 key 的场景

所有 shape 都能走同一套 Kernel 分支，只是 totalLength、tileLength、tailLength 不同。

single strategysame struct

需要 key 的场景

小 M、大 M、不同 dtype、不同可选输入或不同算法路径，需要进入不同 kernel 分支。

shape branchdtype branchworkspace branch
Kernel TilingKey 分支 · conceptual
REGISTER_TILING_DEFAULT(optiling::DefaultTilingData); REGISTER_TILING_FOR_TILINGKEY("TILING_KEY_VAR == 1", optiling::SmallShapeTilingData); REGISTER_TILING_FOR_TILINGKEY("(TILING_KEY_VAR >= 10) && (TILING_KEY_VAR <= 15)", optiling::LargeShapeTilingData); if (TILING_KEY_IS(1)) { GET_TILING_DATA_WITH_STRUCT(optiling::SmallShapeTilingData, tilingData, tiling); // small shape path }
推理场景常见动态维：MatMul 的 M=batch×seq 往往比 K/N 更动态。Decode 时 M 可能小到 1，Prefill 时 M 明显变大，策略通常需要独立评估。

Runtime Decision · 动态 shape 决策视图

动态 shape 场景需要解释运行期分工、合法性边界和策略断点

真实算子不是只使用一组静态 tile 数字。运行时会读取 shape、平台信息和属性，生成 tiling data、tiling key 和 workspace。这里将运行期分工、片上容量、性能断点、芯片路径和测试 shape 扫描放在同一组决策视图中。

两阶段 tiling 模型
硬件边界可在编译期缓存，shape 决策在运行期发生，Kernel 通过 tiling key 进入不同模板路径。
L0AbaseM * baseK * dtypeBytes <= L0AA tile 必须完整放入 L0A。
L0BbaseK * baseN * dtypeBytes <= L0BB tile 必须完整放入 L0B。
L0CbaseM * baseN * 4 <= L0CL0C 按 FP32 4 字节累计结果算。

Step 07 · 验证和调优

验证从正确性开始，调优从断点开始

验证应覆盖 Tiling 的高风险边界：对齐边缘、尾块、key 阈值、Local Memory 容量和 workspace 上限。单一常规 shape 不足以证明切分策略可靠。

测试点暴露的风险开发者动作
最小 shape / decode M=1核利用率低、尾块特殊、策略不适合小 M。构造单独样例，避免仅用大 batch 覆盖。
对齐边缘非 16 或非 data block 对齐。检查 tailLength、mask、padding。
tileNum 边缘最后一个 tile 不满或 loop 次数错误。对照 gmOffset 和输出范围。
key 阈值两侧分支切换造成正确性或性能断崖。阈值前后各测一个 shape。
最大 shapeLocal Memory、workspace、blockDim 上限。查容量、workspace size 和 profiling。
调优顺序：先保证每个断点 shape 正确，再看 profiling；在正确性覆盖尾块和 key 阈值后，再比较 tile 参数的性能。

术语按官方语义使用

术语开发者语义代码落点
Tiling把算子的输入输出按 shape 和硬件能力切成可由 kernel 分块处理的参数。Host 侧 TilingFunc + Kernel 侧解析。
TilingData保存切分算法相关参数的结构体。op_host/*_tiling.h 定义并注册，Host 写入，Kernel 读取。
blockDim核间切分份数，通常与可用 core 和数据规模相关。context->SetBlockDim(...)。
TilingKeyHost 与 Kernel 约定的分支编号，用于选择不同实现路径或结构体。SetTilingKey、TILING_KEY_IS。
workspace设备侧 GM 临时空间，供 kernel 中间结果或辅助数据使用。GetWorkspaceSizes 和 kernel workspace 参数。
blockLength每个 core 负责的数据长度。TilingData 字段，参与 gmOffset 计算。
tileLength每个 core 每次循环处理的数据长度。CopyIn / Compute / CopyOut 的 length。
gmOffset当前 core、当前 tile 在 GM 上的起始位置。blockIdx * blockLength + progress * tileLength。
Tail / Mask无法整除时最后一块的有效元素与屏蔽策略。Kernel 边界判断、mask 参数或 padding 逻辑。

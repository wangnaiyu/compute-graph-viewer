# 训练透视 · 产品需求文档（PRD）

> **工作代号**：TrainScope · 训练透视
> **一句话**：在大模型训练**正在跑**的时候，沿「计算逻辑架构 × 物理多卡」两根轴，实时定位**哪个节点偏差最大**、解释**为什么**、并给出**该怎么调**。
> **版本**：v2 草案（2026-06-16）　**状态**：待评审
> **输入来源**：① 领导双轴定义；② `design-brief-昇腾CANN.md`（13 条用研 insight）；③ `INSIGHT-ANALYSIS.md`（现状对标）；④ 现有 `SPEC.md` 与 TrainScope 实现。
> **证据声明**：用研 13 条 insight 均 `review:unverified / confidence:high`（受访口径，非统计）。本 PRD 据此定方向，关键决策前需对核心支撑页人审。

## 0. 这版改了什么（为什么不再「四不像」）

上一版（SPEC.md / 现有实现）是一个**做得很扎实、但定位错位**的产物。错位只在一个点上：

> **它把产品做成了「一次已知事故的复盘剧本」，而领导和用研要的是「训练过程中的实时偏差监测与即时干预」。**

| 维度 | 上一版（事故复盘剧本） | 本版（实时训练透视） |
|---|---|---|
| 产品隐喻 | 展览馆：答案已写死，用户点着看一遍 | 监护仪：风险未知，帮用户实时发现+决策 |
| 核心动作 | 回放 Step1997 固定七步 | 实时排出「谁偏差最大」+ 根因 + 调整建议 |
| 模型对象 | 盘古 Pro MoE 专用 | **可换**：盘古 / OLMoE / BLOOM…，模型只是接入对象 |
| 主模式 | 单一已烘焙事故 | **实时监护为主**，事故回放为辅 |
| 数据 | 全合成 mock | **真实遥测接入**（逻辑轴可立即换真数据） |
| 缺的那档 P0 | 无跨硬件迁移精度对齐 | 补上（用研定为 P0 横切） |

**三句话的修正**（给领导讲清楚用）：

1. 产品不是"讲清楚一个故障"，是"让开发者自己实时发现故障"——把根因从**预置答案**变成**产品算出来的结论**。
2. 价值不在两根轴各自多漂亮，在**那座桥**：同一个异常在逻辑轴和物理轴上是两个投影，能来回追溯对上号，这才叫"可解释"。
3. "即时调整"不是一句结论就完了，是**给出可操作的下一步**（隔离某卡 / 跳过某步 / 调 lr / 重平衡专家）。

## 1. 产品定义

**电梯陈述**：千亿级大模型训练经常出现「指标崩了、但全机零硬件报错」的情况，工程师要在 TensorBoard 看 loss、profiler 看通信、dump 看权重、config 翻参数之间**人工搬运上下文、靠经验和运气拼因果**——用研里 80–90% 的人无法从日志定位根因。训练透视把这五类证据收进**一屏、一条 step 轴**，沿「逻辑架构 × 物理多卡」两轴实时排出偏差最大的节点、用自然语言直出根因、并给出调整建议，**把数小时的日志考古压缩成一次可视追溯**。

**它是什么**：训练**正确性**的实时可观测 + 可解释定位工具。
**它不是什么**：性能 profiler（不看 kernel 耗时、不接 Chrome Trace）、文档/IDE/社区等生态工具（用研里这些痛点同样高频，但应由并行 initiative 解决，不由本工具兜底）。

---

## 2. 用户与核心场景

**目标用户**（用研 segment）：大模型训练团队、AI 基础设施/算子开发者、运维 Infra。

**核心场景（主线）**：训练正在跑到第 N 步，某个指标开始抖动/漂移。工程师打开训练透视——

1. **偏差雷达**已经把当前偏差最大的几个对象排到了最前（某层 Gate / 某 TP 列卡 / 某条 All-to-All 链路）。
2. 点进去，**两轴联动**：逻辑轴点亮"是 Layer-k 的路由"，物理轴点亮"落在 TP rank2 那一列卡"，两个投影对上号 → 可解释。
3. **根因卡**用一句自然语言给结论 + 证据链（不是抛堆栈）。
4. **调整建议**给可操作的下一步，工程师据此决定动哪个旋钮。

**辅助场景**：训练已经出过事 → **事故回放**模式，沿采集到的数据复盘因果链（复用现有 TrainScope 七步叙事资产）。

**反场景（明确不做）**：性能瓶颈调优、kernel 级 profiling、文档搜索、IDE 插件、算子转写。

## 3. 要解决的问题（每条挂用研事实）

**主攻三条结构性痛点**（用研最高频簇）：

- **根因不可见**：多层穿透报错质量差、真根因被掩盖，**80–90% 用户无法从日志定位根因**。→ 退回 print 调试。`[[调试体验与报错信息]]`
- **工具割裂、人工搬上下文**：系统级与算子级工具数据不互通，"做完性能排查必须人工来回倒数据、字段对齐、时间轴统一、再二次导入"——**用研吐槽最多的点**。`[[开发工具链支持]]`/`[[研发流程自动化]]`
- **单卡观测看不到跨节点**：多节点通信是黑盒，慢节点/跨节点等待难定位，千卡缺自动化诊断、靠人工经验或驻场。`[[多节点集群运维挑战]]`

**外加一条 P0 横切**（SPEC 原本漏掉、用研定为最高优先）：

- **跨硬件迁移精度漂移**：NVIDIA↔Ascend 迁移后 Loss 曲线分叉/漂移、难复现，缺自动化定位 → 人工逐层 Dump 比对。用研明确要"**与架构图关联叠加**"。`[[模型迁移与精度对齐]]`

## 4. 产品理念：双轴 + 一座桥

这是整个产品的灵魂，领导那句话其实给了一个非常干净的双轴定义：

| 轴 | 领导原话 | 承载内容 | 回答的问题 |
|---|---|---|---|
| **计算轴** | 计算过程 / 大模型逻辑架构 | 层、算子、专家、权重、梯度、loss | "是模型结构里**哪个环节**算坏了？" |
| **物理轴** | 物理环境 / 多卡 | NPU/GPU 卡、rank、DP/PP/TP/EP 并行、卡间通信 | "是**哪张卡 / 哪条链路**出的问题？" |

**桥（产品的真正价值）**：

> 同一个异常，在计算轴上表现为「某算子/专家偏差」，在物理轴上表现为「某卡/某通信链路异常」——**它们是同一件事的两个投影**。让开发者在两轴间来回追溯、对上号，这就是"可解释"。

> "可解释" = 能顺一条因果链，从"loss 在涨"走到"是 Layer-k 的 Gate 路由、落在 TP rank2 那列卡、权重下溢成 -inf"——而且这条链是**当前这步算出来的，不是预先写死的**。

**两根支柱**（一切联动从这两件事派生）：
- **统一 step 轴**（时间维度对齐）：所有视图共用一条 step 轴，框选一次 = 全局对齐一次。直接消灭用研最痛的"时间轴靠手动统一"。
- **选中即双向点亮**（对象维度对齐）：点任一对象，两轴其余部分自动点亮关联。关系不靠文档解释，靠点亮自证。

## 5. 核心能力（从领导诉求 × 用研逐条拆解）

> 每条标注：← 来源依据　|　⊙ MVP 还是二期

| # | 能力 | 它做什么 | 来源依据 | 阶段 |
|---|---|---|---|---|
| **C1** | **实时偏差雷达** | 把当前 step 下"偏差最大"的对象（层/卡/链路/专家）自动排序、置顶、告警 | ← 领导"实时检测哪个节点偏差大"；用研"慢节点自动隔离" | ⊙ MVP |
| **C2** | **双轴联动追溯** | 选中任一对象，逻辑轴↔物理轴互相点亮关联，来回对号 | ← 领导"两个维度可解释"；用研"靠运气拼因果" | ⊙ MVP |
| **C3** | **根因直出** | 一句自然语言根因 + 证据链，替代堆栈/日志考古 | ← 用研"报错语义化、直接打印根因而非堆栈、80–90% 定位不了" | ⊙ MVP |
| **C4** | **即时调整建议** | 给可操作下一步：隔离某 rank / 跳过某 step / 调 lr / 重平衡专家 | ← 领导"即时调整"；用研"自动隔离建议、固化专家排查思路" | ⊙ MVP（建议先做规则化，二期再智能） |
| **C5** | **统一观测 + step 轴对齐** | 系统级+算子级+通信+权重一屏，共用 step 轴，框选全局聚焦 | ← 用研"统一观测视图、时间轴自动对齐、一键联动" | ⊙ MVP |
| **C6** | **跨硬件精度对齐（叠加架构图）** | NV↔Ascend 逐算子 relErr、误差源头定位、迁移 Loss 分叉，叠在逻辑轴上 | ← 用研 **P0 横切**"一键张量比对 + 与架构图关联叠加" | 二期（交互骨架已有 ΔD，待接真实 dump） |

**C1「偏差大」如何度量**（关键开放设计，须和算法/Infra 共定）：候选信号 = 该步指标对历史基线的偏离 z-score、梯度范数突变、负载均衡损失骤降、某 rank 利用率/流量对同组均值的偏离。**MVP 先用可解释的阈值/基线规则，不上黑盒模型**（否则违背"可解释"初衷）。

## 6. 两种模式

- **实时监护（主）**：跟随训练 step 推进，偏差雷达 + 根因卡 + 调整建议持续刷新。数据来自实时流式接入（或近实时轮询采集）。
- **事故回放（辅）**：对采集好的历史 run 复盘因果链。**直接复用现有 TrainScope 七步叙事 + 五视图 + 事件总线资产**——它们从"产品主体"降级为"回放模式的内容载体"，一行不浪费。

两模式共用同一套两轴视图与 step 轴，只是数据源（实时流 vs 历史快照）和交互重心（监测告警 vs 复盘追溯）不同。

## 7. 数据策略（真实数据接入）

> 回应"模型对象可换 + 能否用开源真数据"。核心事实：**两根轴的开源数据稀缺度完全不同。**

### 7.1 计算轴遥测 —— 开源充足，可立即换真数据

| 来源 | 提供什么 | 匹配度 | 入口 |
|---|---|---|---|
| **OLMoE**（AllenAI 开源 MoE） | 公开 wandb 训练日志：负载均衡损失、router z-loss、路由饱和/专家专精；256×H100 FSDP | ★★★ 与盘古 MoE 同构，路由坍缩可用真实曲线 | wandb.ai/ai2-llm/olmoe · github.com/allenai/OLMoE |
| **BLOOM / BigScience** | 公开 tensorboard + **人写"训练编年史"**：哪步 loss 尖峰、当时如何干预 | ★★★ 天然就是"发现偏差→即时调整"的真实剧本 | huggingface.co/bigscience/tr11-176B-logs · github.com/bigscience-workshop/bigscience（chronicles.md） |
| **LLM360（Amber / K2）** | wandb 日志 + 系统日志 + 140 checkpoint + 分段数据 | ★★ 360 全透明，含 system log | llm360.ai |
| **OLMo** | 500+ checkpoint + 训练 metrics/日志 | ★★ Dense，可换对象备选 | github.com/allenai/OLMo |
| **openPangu-Ultra-MoE-718B** | 昇腾原生开源 MoE：权重 + 技术报告 | ★ 权重/报告为主，逐步遥测不全 | ai.gitcode.com/ascend-tribe/openpangu-ultra-moe-718b-model |

### 7.2 物理轴遥测 —— 开源极稀缺（诚实约束）

逐 rank 通信流量、All-to-All 字节数、HCCL/NCCL profiler、逐卡利用率这类**物理层 trace 几乎没人公开**。
→ **结论**：逻辑轴可立刻接 OLMoE/BLOOM 真数据；物理轴短期仍需**自采**（在一次自有训练上用 msprof/HCCL dump）或**合成**。这正是 INSIGHT-ANALYSIS 标的 P0「真实数据接入层」的真正难点所在。

### 7.3 接入层架构（让"模型对象可换"成立）

定义一个统一中间 schema，各来源写 adapter 落进去，视图只认 schema：

```
数据源 adapter:  wandb / tensorboard / msprof / HCCL trace / ckpt dump
        │  归一化
        ▼
统一 run schema:  { steps, 计算轴(layers/ops/experts/weights/grad/loss),
                    物理轴(ranks/dp-pp-tp-ep/comm flows/util),
                    桥(对象↔rank↔weight 映射), 基线/异常标注 }
        │
        ▼
两轴视图 + 偏差雷达 + 根因 + 建议（与具体模型解耦）
```

换模型 = 换 adapter + 换拓扑 JSON，视图层不动。盘古/OLMoE/BLOOM 各做一个 adapter 即可。

## 8. 信息架构（复用 PTO workbench-shell，重定义语义）

沿用现有两层分栏外壳，但每个区域的**语义从"复盘剧本的一幕"改为"实时监护的一个面"**：

```
┌─ 顶部：run 标识 + 实时状态条（当前 step / 偏差雷达 TOP-N 告警）────────────┐
├──────────────┬─────────────────────────────────┬────────────────────┤
│ 偏差雷达 +    │  计算轴（逻辑架构图）[主舞台]      │ 物理轴 / 根因 +     │
│ 参数信号      │  层/算子/专家，偏差节点高亮       │ 调整建议            │
│ [左 rail]    │  ← 选中双向点亮 →                │ [右 inspector]      │
├──────────────┴─────────────────────────────────┴────────────────────┤
│ 物理轴（多卡 mesh）+ 效果曲线 [底部 dock · 共用 step 轴 scrubber]        │
│ 卡/链路流量、黑洞、逐 rank 偏离　|　train/val loss · eval，可框选          │
└────────────────────────────────────────────────────────────────────┘
```

复用清单见 §10。视觉 100% 走 PTO token，至多一层可见边界，图节点无阴影（沿用现有规范）。

## 9. MVP 范围

**MVP（实时监护跑通所必需）**：
- F1 偏差雷达（C1）：基线/阈值规则排出 TOP-N 偏差对象，告警置顶。
- F2 两轴视图 + 双向点亮（C2、C5）：逻辑架构图 + 多卡 mesh，共用 step 轴，选中互点亮。
- F3 根因卡（C3）：自然语言根因 + 证据链。
- F4 调整建议（C4，规则化版）：针对已识别异常给可操作下一步。
- F5 数据接入层（§7.3）：至少接通 1 个真实逻辑轴来源（推荐 OLMoE）+ 1 套物理轴（自采或合成）。
- F6 事故回放模式（复用现有资产）。

**二期**：
- C6 跨硬件精度对齐接真实 dump（P0 横切，最高优先）。
- 多故障场景（TP AllReduce 掉队 / PP 气泡）。
- 调整建议从规则化升级为"固化专家排查思路"的智能诊断。
- 真整合 msprof + cann_sim + 通信 trace（产品工程，非可视化器单独能给）。

> **二期排序净校准**（据 INSIGHT-ANALYSIS §8.3）：把**跨硬件迁移提到最前**（用研依据最强、SPEC 原本没有），**显存维度往后挪**（用研里非高频痛点）。

## 11. 风险与开放问题

1. **"偏差大"的度量定义**（§5）：阈值/基线规则需和算法、Infra 共定，且必须**可解释**，不上黑盒。— 待定，最高优先。
2. **物理轴真实数据缺口**（§7.2）：逐 rank 通信 trace 开源拿不到，MVP 物理轴大概率仍需自采或合成。— 需决定首发用哪种。
3. **实时性的工程代价**：真·流式接入需要训练侧埋点 + 近实时管道；若一期落不了，可先做"近实时轮询采集"过渡。— 需和 Infra 评估。
4. **产品边界 vs 总目标**：本工具对"昇腾好不好用"是**必要但不充分**的一块拼图，生态侧痛点（文档/IDE/算子/社区）须并行 initiative 解决，别指望本工具兜底。
5. **证据可靠性**：13 条 insight 全 `unverified`、源偏 NV 视角、量化数字是口径非统计。重投入前应对核心支撑页人审升 `human-checked`。

---

## 12. 成功标准

1. 训练**正在跑**时，偏差雷达能自动排出 TOP-N 偏差对象，无需翻日志。
2. 选中任一对象，计算轴与物理轴**两个投影同时点亮、能对上号**（可解释成立）。
3. 根因以**一句自然语言 + 证据链**呈现，而非堆栈。
4. 每个已识别异常都给出**至少一条可操作的调整建议**。
5. 至少接通**一个真实逻辑轴来源**（OLMoE/BLOOM），模型对象可经 adapter 替换。
6. 事故回放模式下，现有七步因果链仍 1:1 可用（资产零浪费）。

> 取自 HF `FreedomIntelligence/openPangu-Ultra-MoE-718B` 的 `config.json`（真实值，替代 DeepSeek 代理）。

**架构**：61 层（前 `first_k_dense_replace=3` 层 dense + 58 层 MoE）+ 1 个 MTP 头（`num_nextn_predict_layers=1`）；hidden 7680，intermediate 18432，moe_intermediate 2048。
**MoE**：256 路由专家 + 1 共享专家（`n_shared_experts=1`），top-8 激活（`num_experts_per_tok=8`），`routed_scaling_factor=2.5`，`norm_topk_prob=true`。共享专家恒激活 → 路由坍缩只影响 256 路由专家，不影响共享专家。
**注意力（MLA）**：128 头，`q_lora_rank=1536`，`kv_lora_rank=512`，`qk_nope_head_dim=128` + `qk_rope_head_dim=64`，`v_head_dim=128`。
**其它**：vocab 153600，max_position 131072，`sandwich_norm=true`，`rope_theta=2.56e7`。
**规模/训练**：718B 总 / 39B 活；19T tokens；实测 **4000×昇腾 910B**（64GB HBM），MFU 18.9%，0.61M tok/s。

**派生的并行/渲染网格**（满足约束 + 对标 DeepSeek-V3 + 显存自洽的可信构造；精确切分 openPangu 未全公开，"4000"约为 4096=2¹²）：

*公式陷阱*：`N = TP × PP × EP × DP × CP` 会把 EP 重复计数。真实关系：**物理卡总数 `N = TP × PP × CP × DP`**（这四个唯一定位"哪张卡"）；专家层再切——256 专家分散到 EP 个 rank，**`EP ⊆ DP`**，每卡持有专家 = 256/(PP×EP)。

*EP 的显存下界*（关键约束）：256 路由专家优化器状态 ≈ 10.9TB（680B×16B），每卡仅 1/(PP×EP)。要塞进 64GB（留 ~40GB 给专家）：`10.9TB/(16×EP) < 40GB → EP ≥ 32`。EP=16 每卡 ~42GB 会爆，EP=32 每卡 ~21GB 可行。**EP 不是自由选，有硬地板。**

*推荐配置 ①（昇腾友好）*：

| TP | PP | CP | DP | EP | = N | 每卡负载 |
|---|---|---|---|---|---|---|
| 8（锁节点内 HCCS） | 16（≈4 层/段） | 1 | 32 | 32（住在 DP 内） | **4096** | 4 层 + 1/8 张量 + 8 个专家 |

*渲染切片*：物理全集（16 段流水 × 32 DP 副本 × 8 TP）不画；**画 1 个 EP 组 = 32 张卡**，256 专家摊在上面（每卡 8 个），All-to-All 在组内流动，排 **4×8** 网格，黑洞 = 其中 1 张 EP-rank（8 个专家被饿死），共享专家不受影响。现有 SPEC 的 8×4 网格正好就是一个 EP=32 组。
*流量标度*：All-to-All 每 token 字节 ≈ 激活路由数 × hidden(7680) × bytes，作物理轴流量基准。

## 附录 B：openPangu 缺训练落盘数据 → loss 监控场景怎么落

**事实**：openPangu 公开的是**权重 + config + 技术报告**，**没有公开逐步训练遥测**（per-step loss/grad/路由日志）。所以没有可直接监控的 openPangu 真实 loss 时序。

**数据策略（诚实）**：loss 监控场景的**动态信号**借真实开源 MoE 训练——OLMoE（路由/负载均衡/router z-loss 的真实逐步曲线）+ BLOOM 训练编年史（真实 loss 尖峰与人工干预节奏）——重投影到 openPangu 架构上；**绝对标度合成，相对形态真实**。要拿到 openPangu 自己的真值，只能在一次自有 Ascend run 上埋点采集（产品化阶段，非 demo）。

**场景设计（与数据源无关）**：loss 监控 ≠ 只看 loss，而是 **loss + 其前导指标共用一条 step 轴**：
1. **基线带**：对 loss 取滚动 EMA + 方差带，越带即偏差（z-score）= "偏差大"的可解释判据，不上黑盒。
2. **前导指标**：grad norm、负载均衡损失、router z-loss、专家负载方差——它们常**先于** loss 崩动。偏差雷达盯的是"谁先偏、偏得最大"。
3. **监控→诊断交接**：loss 越带 → 看哪个前导指标先偏 → 指向轴（负载均衡损失骤降 = 路由问题 = 逻辑轴 Gate；某 PP stage grad 尖峰 = 物理轴）→ 落到 C1→C3。

## 附录 C：术语对照（大白话）

用一个「256 人会诊中心」的比喻把核心术语串起来，便于对非计算背景的同事讲清楚。

| 术语 | 大白话 | 比喻（会诊中心） |
|---|---|---|
| **专家 / Expert** | 模型每层里的一个子网络，共 256 个 | 256 个专科医生 |
| **路由器 / Gate（门控）** | 决定每段文字交给哪几个专家处理 | 门口的分诊台 |
| **top-8 激活** | 每个 token 只请 256 个里最对口的 8 个专家 | 每个病人只挂 8 个对口医生 |
| **共享专家** | 1 个恒定参与的专家，不受路由影响 | 每个病人都先过的全科医生 |
| **负载均衡损失** | 逼路由器把活摊匀的指标；**骤降=路由失控的前导警报** | 防止分诊台偏心的考核 |
| **专家被饿死 / Expert starved** | 路由器不再给某些专家派 token → 收不到梯度、停止学习、恶性循环 | 某些医生一个病人都分不到，技能荒废 |
| **专家并行 / EP** | 256 专家太大一张卡装不下，摊到多张卡上（EP=32 → 每卡 8 个） | 256 个医生分散到 32 间诊室，每间 8 个 |
| **All-to-All 通信** | 把每个 token 通过网络送到持有它要的专家的那张卡 | 把病人送到对口医生所在的诊室 |
| **黑洞 / Black hole** | 被饿死的专家所在的卡，流入 token 枯竭 → 物理视图上流量线干瘪 | 没人要看的那间诊室，门可罗雀 |

**一个故障，三种说法（产品要让它们自动对上号 = 可解释）**：
- 逻辑视角：路由坍缩、专家被饿死
- 监控视角：负载均衡损失骤降到 0
- 物理视角：某张卡变成流量黑洞

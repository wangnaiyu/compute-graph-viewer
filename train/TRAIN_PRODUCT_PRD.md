# PTO Train 产品方向探索 PRD 初稿

## 1. 背景

`train` 项目前期已经形成两条重要洞察：

- 学习页已经能把 Qwen7B、Qwen3、Qwen3-MoE、DeepSeek-V3.2、Ascend/MindSpeed 生态、脚本参数和架构图放在同一个学习工作台里。
- `Training digital twin demo · HTML.txt` 展示了另一种形态：围绕一次训练 run，把 step、loss、MFU、吞吐、硬件热力、系统通信、故障风险、数据管道、事件流和 what-if 推演放在一个“训练孪生体控制台”里。

下一阶段产品不应只继续扩充知识点，也不应直接变成传统监控大屏。更有价值的方向是：

> 帮助用户从“看懂模型和脚本”过渡到“看懂一次真实训练为什么能跑、为什么失败、下一步该怎么改”。

## 2. 目标用户

| 用户 | 当前痛点 | 需要的产品能力 |
| --- | --- | --- |
| 初学者 / 新同事 | 能看懂单个术语，但不知道模型、脚本、数据、权重、日志如何连起来。 | 分层学习路径、架构到脚本映射、字段注释、启动前检查。 |
| 模型训练开发者 | 配置项分散在官方文档、shell 脚本、模型 config、日志中，排错成本高。 | 脚本体检、参数一致性检查、运行阶段解释、失败归因。 |
| 性能/系统调优人员 | profiling、通信、显存、dataloader、checkpoint 信息碎片化。 | 资源与质量同轴时间线、风险因子、straggler 和通信瓶颈解释。 |
| PM / demo 讲解者 | 需要把复杂训练系统讲成一条可理解的故事。 | 训练流程叙事、可演示的孪生体、实验结论卡。 |

## 3. 产品定位

产品暂定名：**Training Developer Twin**

一句话定位：

> 面向 Ascend/MindSpeed 大模型训练开发的学习、体检、运行解释和实验决策工作台。

核心差异：

- 不是 TensorBoard 替代品：TensorBoard 强在基础指标、模型图和 profiling，本产品强在解释链和下一步动作。
- 不是纯文档站：每个概念都要能回到模型文件、脚本字段、运行日志或 profiling 证据。
- 不是大而全 MLOps：第一阶段只围绕训练开发者从脚本到 run 的关键断点。

## 4. 产品原则

- 证据优先：官方文档、本地 Qwen7B 源码、MindSpeed-LLM 脚本、日志和 profiling 结果必须可追溯。
- 学习到实战连续：同一套概念从 Qwen7B 入门、Qwen3 快速训练、MoE/DeepSeek 工程复杂度逐级展开。
- 解释优先于堆图：每张图都回答“为什么这个值重要、异常可能来自哪里、下一步怎么验证”。
- 决策优先于记录：不仅保存曲线，还要保存实验假设、diff、结论和是否继续。
- PTO 设计系统一致：顶部 header 保持透明、页面无额外 header 下方外边距，正文最小字号 14px，卡片化内容服务阅读而不是装饰。

## 5. 方案探索

### 方案 A：训练学习工作台

定位：把当前 `index.html` 和 `training-mental-model.html` 产品化成“从模型源码到训练脚本”的学习路径。

适合目标：

- 新用户第一次学习 Ascend/MindSpeed LLM 训练。
- 讲解 Qwen7B、Qwen3、MoE、DeepSeek 的结构差异。
- 在没有真实训练 run 数据时也能交付价值。

核心功能：

- 模型案例路径：Qwen7B -> Qwen3-8B -> Qwen3-MoE -> DeepSeek-V3.2 -> GRPO。
- 架构图联动：点击 config 字段，高亮 Embedding、Attention、MLP、LM Head、MoE Router 等节点。
- 脚本字段解释：SEQ_LENGTH、TP、PP、EP、CP、MBS、GBS、DATA_PATH、TOKENIZER_PATH、CHECKPOINT_PATH 都有“为什么重要”和“改错会怎样”。
- 初学者类比解释：继续保留“做饭类比”，但收敛为辅助标注，不占据主内容。
- 学习检查点：每个楼层最后给出 3 到 5 个必须回答的问题。

Wireframe：

```text
┌────────────────────────────────────────────────────────────────────┐
│ PTO Train · Ascend Training Developer Experience                    │
├──────────────────────────────┬─────────────────────────────────────┤
│ 学习路径 Tabs                 │ 架构/脚本证据图                     │
│ [生态][Qwen7B][Qwen3][MoE]   │ ┌─────────────────────────────────┐ │
├──────────────────────────────┤ │ Qwen7B / Qwen3 / DeepSeek Graph  │ │
│ 楼层正文                      │ │ 点击节点后显示参数来源           │ │
│ ┌──────────────────────────┐ │ └─────────────────────────────────┘ │
│ │ 1. 概念解释               │ │ 证据卡                             │
│ │ 2. 官方/本地来源           │ │ ┌────────┐ ┌────────┐ ┌────────┐    │
│ │ 3. 参数注释卡              │ │ │config  │ │script  │ │doc     │    │
│ │ 4. 常见错误                │ │ └────────┘ └────────┘ └────────┘    │
│ └──────────────────────────┘ │                                     │
│ 检查点：我是否能解释这个字段？ │                                     │
└──────────────────────────────┴─────────────────────────────────────┘
```

优点：

- 最贴近当前项目资产，落地最快。
- 对内容质量要求高，但对真实 telemetry 依赖低。
- 能先把 Qwen7B 内容做扎实，适合作为用户入口。

风险：

- 如果只做学习页，产品感可能偏“教程”，差异化不够强。
- 很难证明能降低真实训练失败率，需要引入脚本体检或 run 解释能力。

### 方案 B：训练孪生体运行控制台

定位：参考 `Training digital twin demo · HTML.txt`，把一次训练 run 做成可解释的实时控制台。

适合目标：

- 用户已经能启动训练，但不知道如何判断训练是否健康。
- Demo 需要快速展示“训练系统正在发生什么”。
- 后续接入日志、profiling、msTT、NPU telemetry、事件流。

核心功能：

- Run 摘要：step、loss、MFU、tokens/s、tokens seen、ETA、checkpoint 状态。
- 训练动态：loss、val loss、grad norm、learning rate、MFU 曲线。
- 硬件基础设施：NPU/节点热力、HBM、利用率、异常卡、straggler。
- 软件运行层：TP/PP/DP/EP/CP、通信耗时、pipeline bubble、dataloader wait。
- 风险预测：未来 30 分钟中断概率，归因到热、OOM、通信、loss spike、I/O、checkpoint。
- 数据管道：数据混合比例、加载吞吐、序列长度分布。
- What-if 推演：调整 TP/PP/DP/MBS/GBS/recompute，预测吞吐、显存、bubble、成本和 ETA。

Wireframe：

```text
┌────────────────────────────────────────────────────────────────────┐
│ 训练孪生体控制台  run atlas-qwen3-8b-r12  TRAINING  sync 12ms       │
├──────────┬──────────┬──────────┬──────────┬──────────┬──────────┤
│ Step     │ Loss     │ MFU      │ tok/s    │ Seen     │ ETA      │
├──────────────────────────────────────┬─────────────────────────────┤
│ 训练动态                             │ 风险预测                    │
│ ┌──────────────────────────────────┐ │ ┌─────────────────────────┐ │
│ │ loss / val / grad / lr / MFU     │ │ │ 31% 中风险              │ │
│ └──────────────────────────────────┘ │ │ 热/通信/straggler 因子   │ │
├──────────────────────────────────────┤ └─────────────────────────┘ │
│ 硬件热力图                           │ 事件流                      │
│ □□□□ □□□□ □□□□ □□□□                 │ 03:12 checkpoint done       │
│ □□□□ □□□□ □□□□ ■□□□                 │ 03:13 node-7 temp warning   │
├──────────────────────────────────────┼─────────────────────────────┤
│ 系统/通信/显存条                      │ 数据管道                    │
│ HBM 78%  NCCL 11ms  bubble 18%       │ Web 43% Code 22% Math 6%    │
├──────────────────────────────────────┴─────────────────────────────┤
│ What-if: TP [4] PP [4] DP [32] MBS [2] GBS [128] recompute [on]     │
│ 输出：MFU +4%  ETA -9h  HBM +6GB  风险 +12%                         │
└────────────────────────────────────────────────────────────────────┘
```

优点：

- 产品感最强，容易演示。
- 能把官方 profiling、日志和运行指标转成解释型体验。
- 后续可连接真实训练系统，形成更高门槛。

风险：

- 第一版如果没有真实数据，只能是模拟 demo，容易被看作“漂亮大屏”。
- 指标来源、采样频率、权限、日志格式、run 采集成本都需要工程投入。
- 风险预测必须先用规则引擎表达，不宜过早承诺 ML 预测准确性。

### 方案 C：实验决策中心

定位：超越“看曲线”，让用户判断一次训练实验是否值得继续、回滚、扩容或进入候选模型。

适合目标：

- 用户已经有多次 run，需要比较不同 config、数据、checkpoint 和 eval。
- 希望从 TensorBoard 风格标量曲线升级为“实验结论层”。
- 面向模型研发团队协作和复盘。

核心功能：

- 实验决策卡：假设、baseline、candidate、config diff、指标变化、成本变化、结论和下一步。
- 多目标 trade-off：质量、吞吐、显存、延迟、成本、稳定性同屏对比。
- 样本级下钻：从整体指标点到具体 prompt、label、prediction、reward、judge 结果。
- 训练阶段时间轴：warmup、稳定下降、plateau、loss spike、checkpoint、数据切换、eval。
- Lineage 图：dataset、tokenizer、code commit、config、checkpoint、eval、model candidate。
- 报告层：自动生成本周实验摘要、通过/拒绝原因和待复查 slice。

Wireframe：

```text
┌────────────────────────────────────────────────────────────────────┐
│ 实验决策中心  Project qwen3-sft  Baseline run-041  Candidate run-058 │
├──────────────────────────────┬─────────────────────────────────────┤
│ 实验卡列表                    │ 决策详情                            │
│ ┌──────────────────────────┐ │ ┌─────────────────────────────────┐ │
│ │ run-058 质量 +0.7%        │ │ │ 假设：增大 GBS + 延长 warmup      │ │
│ │ 成本 +18%  不建议继续      │ │ │ 结论：质量提升小，long slice 退化 │ │
│ └──────────────────────────┘ │ └─────────────────────────────────┘ │
│ ┌──────────────────────────┐ │ Trade-off Map                       │
│ │ run-061 吞吐 +11%         │ │ quality ↑                           │
│ └──────────────────────────┘ │       ● run-058                      │
├──────────────────────────────┤ latency/cost →                       │
│ 筛选：模型/数据/日期/状态       │                                     │
├──────────────────────────────┴─────────────────────────────────────┤
│ Config Diff | Metric Diff | Sample Drilldown | Lineage | Report     │
│ DATA_PATH v12 -> v13                                                │
│ lr 3e-4 -> 2e-4                                                     │
│ hard_long_context F1 -4.2                                           │
└────────────────────────────────────────────────────────────────────┘
```

优点：

- 差异化最清晰，能解释“为什么不仅是 TensorBoard”。
- 对团队协作有长期价值。
- 可以从静态样例和本地 JSON/CSV 先做，不必一开始接实时系统。

风险：

- 需要明确 eval 数据结构，否则样本级下钻会空泛。
- 对用户已有实验记录的规范性有要求。
- 如果没有训练运行解释，可能离 Ascend/MindSpeed 训练开发场景稍远。

## 6. 方案对比

| 维度 | 方案 A 学习工作台 | 方案 B 训练孪生体 | 方案 C 实验决策中心 |
| --- | --- | --- | --- |
| 当前资产复用 | 高 | 中 | 中 |
| 第一版落地速度 | 快 | 中 | 中 |
| 产品差异化 | 中 | 高 | 高 |
| 对真实数据依赖 | 低 | 高 | 中 |
| 对新手价值 | 高 | 中 | 中 |
| 对训练工程价值 | 中 | 高 | 高 |
| Demo 冲击力 | 中 | 高 | 中 |
| 长期平台价值 | 中 | 高 | 高 |

建议决策：

> 第一阶段选择 **A + B 的轻量组合**：用学习工作台做入口，用“模拟/半真实训练孪生体”承接训练运行解释。方案 C 作为第二阶段差异化能力，在有 run/eval 数据后补上。

原因：

- 只做 A，容易停留在教程。
- 只做 B，真实数据接入成本高，而且新手不知道这些指标为什么重要。
- 只做 C，需要 eval 和 run lineage 基础，当前还不到最合适的起点。
- A+B 可以形成完整故事：先看懂 Qwen7B/Qwen3 脚本，再看一次训练 run 如何健康运行或失败。

## 7. 推荐 MVP

MVP 名称：**Training Run Explainable Twin**

MVP 目标：

> 用户输入一个模型训练场景后，能得到脚本体检、架构解释、运行状态解释和下一步建议。

### 7.1 P0 功能

| 模块 | 输入 | 输出 | 验收标准 |
| --- | --- | --- | --- |
| 模型路径选择 | Qwen7B、Qwen3-8B、Qwen3-MoE、DeepSeek-V3.2 | 学习路径、架构图、关键参数 | 用户能解释模型层数、hidden size、attention heads、seq length、tokenizer、LM Head。 |
| 脚本体检器 | shell/yaml/config 片段 | 路径、并行、数据、权重、rank、日志风险 | 能拦截 DATA_PATH/TOKENIZER_PATH 缺失、TP/PP 与节点数不一致、日志目录缺失。 |
| 架构到脚本映射 | config + training args | 参数卡、模块高亮、影响说明 | 点击 SEQ_LENGTH/TP/PP/MBS/GBS 能看到影响显存、通信、吞吐还是收敛。 |
| 训练孪生体 demo | 模拟 run 或样例日志 | run vitals、训练动态、硬件/系统/数据状态、事件流 | 用户能在 3 分钟内判断 run 是否健康，以及最大风险来自哪里。 |
| What-if 推演 | TP/PP/DP/MBS/GBS/recompute | MFU、吞吐、显存风险、ETA、成本变化 | 每个输出都有“方向性解释”，不只给数字。 |
| 结论卡 | 当前脚本 + run 状态 | 建议继续/暂停/改配置/收集 profiling | 用户能把页面结果转成下一步动作。 |

### 7.2 P1 功能

- 接入真实日志解析：MindSpeed-LLM 训练日志、checkpoint 日志、OOM/NaN/HCCL 错误。
- 接入 profiling 摘要：通信占比、算子耗时、内存占用、pipeline bubble。
- 接入 eval 结果：指标 diff、样本下钻、失败 slice。
- 生成实验报告：自动汇总本次训练配置、风险、异常、结论和复现信息。

### 7.3 暂不做

- 不做完整 MLOps 平台。
- 不做权限、队列、资源调度系统。
- 不承诺真实 ML 风险预测，第一版使用规则引擎和可解释阈值。
- 不直接支持所有模型，优先把 Qwen7B/Qwen3/DeepSeek 路径做深。

## 10. 成功指标

| 阶段 | 指标 |
| --- | --- |
| 学习 | 新用户 10 分钟内能解释 Qwen7B 的 5 个关键结构参数。 |
| 启动前 | 脚本体检能识别 80% 以上常见 P0 配置错误。 |
| 运行中 | 用户能在 3 分钟内判断 run 健康、风险来源和下一步动作。 |
| 复盘 | 每次实验自动生成结论卡，减少人工整理实验记录时间。 |
| 产品 | 用户愿意把 train 页面作为训练前 checklist 和训练中解释入口，而不只是阅读材料。 |

## 11. 数据需求

P0 可用静态/半静态数据：

- 本地 Qwen7B README、config、generation_config、modeling_qwen.py、safetensors index。
- MindSpeed-LLM Qwen3、Qwen3-MoE、DeepSeek 示例脚本。
- 官方 Ascend/MindSpeed 文档摘录和链接。
- `Training digital twin demo · HTML.txt` 中的模拟 run 指标。

P1 需要接入：

- MindSpeed-LLM 训练日志。
- profiling/msTT 摘要。
- checkpoint 元数据。
- eval JSON/CSV。
- 样本级 prediction 表。

## 12. 关键风险

- 内容风险：如果官方来源和本地来源没有校准，产品会变成 AI 生成教程，可信度不够。
- 数据风险：如果没有真实日志和 profiling，训练孪生体只能是 demo，不能成为工作流。
- 复杂度风险：同时做学习、监控、实验管理会发散，必须用 Qwen7B/Qwen3/DeepSeek 三条路径收敛。
- 解释风险：what-if 推演必须明确是启发式估算，不能伪装成精确性能模型。

## 13. 需要你决策的问题

1. 第一版主入口选哪个？
   - A：学习路径入口，适合继续打磨 train 当前页面。
   - B：训练孪生体入口，适合做更强 demo 和运行解释。
   - A+B：推荐。学习入口解释概念，孪生体入口解释运行。

Design Translation

每个 PTO 模块都要回答同一个问题：用户看到一个结果时，能不能追溯它来自哪个官方概念、哪个输入约束、哪个编译阶段、哪个硬件路径、哪条 profiling 证据。

编译证据Pass、PATH、snapshot、graph diff。
参数证据TilingData、blockDim、workspace。
硬件证据L1/L0/UB、MTE route、Cube/Vector。
时间证据step、stream、task、wait、overlap。
修复证据issue、action、verify metric。

METHOD

PTO 的设计策略是把工程判断转化为可追溯证据

后续产品化不按页面数量推进，而按“工程判断、官方锚点、证据对象、交互机制、复用模式”组织能力沉淀。
05 / 16
Definition
设计策略 = 用户判断 + 官方锚点 + 证据对象 + 交互机制 + 复用模式

每一项核心能力都必须回答：用户在做什么工程判断，它对应哪个官方概念，证据从哪里来，交互如何帮助定位，最终如何沉淀为 PTO pattern 或跨模块范式。

A从模块到机制Pass-IR 的价值不是图查看器，而是编译时间线驱动 graph snapshot。
B从视觉到证据工作台视觉必须服务定位、解释、修复和验证，而不是停留在展示效果。
C从单页到复用关键图元、面板和交互需要沉淀到 PTO patterns 与组件体系。
D从原型到产品现有页面通过统一数据和联动协议，收束为稳定的 Evidence Workbench。

EVIDENCE LAYER 01

Pass-IR 把 pass folder 变成编译时间线

图编译证据是整条链路的源头：先解释图如何变化，再进入参数、执行和诊断。
07 / 16
01 · compile evidence

编译过程需要可导航

Pass-IR 把 compile_debug_mode 输出目录转成阶段导航。用户通过 pass、PATH、snapshot 和 locked flow 追溯图结构变化。

Pass timeline 替代文件树，降低编译阶段定位成本。
Locked Flow 把节点上下游从大图中提炼出来。
Semantic、latency、subgraph 颜色模式服务不同证据切换。
产品价值：让编译阶段从黑箱输出变成可导航过程，为后续执行、内存和 Profiling 证据提供源头索引。

Tiling 从参数结果变成因果解释

02 · tiling causality

官方 Tiling 把 shape/attr 转成 TilingData、blockDim、TilingKey 和 workspace。PTO 将这条因果链显性化，避免参数解释停留在经验判断。

把输入约束、参数设计、kernel 消费拆成连续叙事。
3D tensor 视觉服务参数理解，而不是替代因果解释。
参数解释 panel 可扩展到更多算子开发场景。
产品价值：降低算子开发者理解和调整 Tiling 参数的成本，使参数选择从经验判断转为可解释推导。

EVIDENCE LAYER 05

Swimlane 让执行任务可以对比、下钻和回到源码

内存证据之后，运行任务需要进入时间维度：对比、Diff、下钻和源码联动。
11 / 16
05 · execution evidence

把时间线变成可操作证据

Swimlane 的价值不是展示任务条，而是目录导入、Before/After、Diff、Program 绑定和 Source Flow 联动，让 task 能进入上下游证据。

资源管理面板收口所有本地数据绑定，减少状态分散。
对比和 Diff 服务性能回归定位，而不是装饰性双图。
深入任务卡片把 task 导向 Pass-IR 和 Source Flow。
产品价值：让一次执行任务能够回到 IR、源码和前后对比，形成时间维度的证据层。

06 · profiling diagnosis

报告不再是终点

Graph Evidence Workbench 把性能报告中的问题节点、trace task、Inspector 诊断和图结构联动起来，让用户从结论回到证据。

Graph node click、swimlane task click、Inspector mapped node 是同一个 selection state。
priority filter 让 P0/P1/P2 从报告优先级变成图上可见层。

产品价值：将性能报告从静态结论升级为可定位、可下钻、可验证的诊断工作台。

同一类交互在不同模块保持同一形状和状态语言。

产品价值：提升跨模块连续性，降低用户在不同工具之间重新学习界面的成本。
